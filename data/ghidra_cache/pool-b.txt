INTERPRETATION - helpers are a state-transition queue, NOT add/remove
--------------------------------------------------------------------------------
The two helpers are NOT "add_task" and "remove_task" in the "register /
unregister for per-frame ticking" sense. They are state-transition helpers
that push deferred state-change commands for resource-like objects:

  FUN_7103559340 = schedule_transition_to_state_0(task)   // "activate"
  FUN_7103559220 = schedule_transition_to_state_1(task)   // "deactivate"

Evidence:
  - The vt[0x38] gate is inverted between the two: add rejects state==0 tasks,
    remove rejects state!=0 tasks. Textbook "no-op if already in target state".
  - The vt[0x40] post-call passes the target state byte (0 or 1) matching the
    queue entry op_byte.
  - The tiny toggle helper FUN_7103579a60 exposes this directly:
        if (enable_bit)  FUN_7103559220();   // to state 1 = disable
        else             FUN_7103559340();   // to state 0 = enable
  - Singleton uses recursive_mutex (not plain mutex), consistent with resource
    loads spawning sub-loads that re-enter the queue on the same thread.

Task vtable slots inferred:
  vt[0x10] = destructor (shared_ptr refcount-0 path)
  vt[0x20] = "apply state 1" (commit deactivation)         (seen in main_loop)
  vt[0x28] = "apply state 0" (commit activation)           (seen in main_loop)
  vt[0x38] = get_state()           (predicate; != 0 when "live")
  vt[0x40] = set_pending_state(u8) (side channel helper<->drain)
  vt[0xa8] = prepare_activation(u8) (only called by FUN_7103559340)
  vt[0x340] = additional pre-activation hook (huge offset -> deep class)

Vtable is therefore at least 0x348 bytes wide (~0x69 entries). That is a
very large base class - consistent with a cross2app "Resource" or
"Renderable" abstract base, not a small sim-task interface.
